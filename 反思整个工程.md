


# 这个工程确实 严重依赖提示词 ，提示词占比约 15.3%。

## 反思：严重依赖提示词，封装华为套件 deepdiver_v2

结论 ：提示词占比约 15.3%，这确实是一个 相当高的比例 ！

### 2. 提示词的核心作用
让我分析一下提示词在系统中的关键作用：

1. 角色定义 ：明确智能体的职责和定位
2. 工作流程 ：定义完整的任务处理流程
3. 工具描述 ：详细列出可用工具及其参数
4. 协议规范 ：规定任务分解、并行执行、文档管理等规则
5. 边界条件 ：明确何时调用哪些子智能体
6. 完成标准 ：定义任务成功的判断标准
### 3. 提示词依赖的深层原因分析 A. 为什么这么依赖提示词？
1. 控制智能体行为 ：
   
   - 没有提示词，LLM 无法知道自己的角色
   - 不知道应该调用哪些工具
   - 不知道何时停止任务
   - 不知道如何与其他智能体协作
2. 华为的架构设计理念 ：
   
   - 提示词是"软约束"，而非"硬编码"
   - 通过自然语言引导智能体行为
   - 便于快速调整和迭代优化
3. Multi-Agent 系统的复杂性 ：
   
   - 需要协调多个智能体
   - 需要处理复杂的任务依赖关系
   - 需要处理并行执行和冲突解决
   - 这些都需要通过提示词来规范 B. 这种方式的优缺点
优点 ：

- ✅ 灵活性高 ：可以通过修改提示词快速调整智能体行为
- ✅ 可维护性强 ：逻辑集中在提示词中，易于理解和修改
- ✅ 适应性强 ：可以针对不同场景定制不同的提示词
- ✅ 快速迭代 ：不需要修改代码，只需调整提示词
缺点 ：

- ⚠️ 提示词工程复杂 ：需要精心设计提示词，否则智能体行为不稳定
- ⚠️ 依赖 LLM 能力 ：提示词的效果完全依赖于 LLM 的理解能力
- ⚠️ 调试困难 ：当智能体行为异常时，很难确定是代码问题还是提示词问题
- ⚠️ 提示词膨胀 ：随着功能增加，提示词会越来越长，难以维护


### 核心原因 ：

1. 华为的架构设计理念 ：通过提示词而非硬编码来控制智能体行为
2. Multi-Agent 系统的复杂性 ：需要通过提示词协调多个智能体的协作
3. 灵活性和可维护性 ：提示词可以快速调整，不需要修改代码
这种方式的本质 ：

- 将"智能体逻辑"从代码中转移到提示词中
- 通过自然语言引导 LLM 自主决策
- 提示词成为系统的"软约束"和"行为指南"
优缺点 ：

- ✅ 优点：灵活性高、可维护性强、适应性强、快速迭代
- ⚠️ 缺点：提示词工程复杂、依赖 LLM 能力、调试困难、提示词膨胀
这是一个典型的**"Prompt Engineering + Multi-Agent"**架构设计！

### 关键特点 A. 每个独立的提示词
- ✅ PlannerAgent ：369 行，负责任务规划和协调
- ✅ WriterAgent ：101 行，负责内容生成和写作
- ✅ InfoSeeker ：51 行，负责信息搜索和研究 B. 提示词不会被拆分
- ❌ 不会 将 PlannerAgent 的提示词拆分后发送给其他 Agent
- ❌ 不会 在 Agent 之间共享提示词
- ✅ 每个 Agent 都有自己独立的完整提示词
- ✅ 每个 Agent 执行时都会独立调用 LLM C. Agent 之间的协作方式
1. 通过任务分配 ：PlannerAgent 通过调用 assign_subjective_task_to_writer 等方法分配任务
2. 通过共享资源 ：通过共享 MCP 客户端和工作空间
3. 通过结果返回 ：子 Agent 执行完成后返回结果给 PlannerAgent
4. 通过对话历史 ：每个 Agent 维护自己的对话历史，不会共享
### 5. 总结
回答您的问题 ：

1. 复杂的提示词，直接送到模型中吗？
   
   - ✅ 是的 ，每个 Agent 的完整提示词都会直接发送给 LLM
   - ✅ 包含完整的系统提示词（system_prompt）和用户消息（user_message）
2. 多 Agent 协作的时候，这些提示词会做拆分吗？
   
   - ❌ 不会拆分 ，每个 Agent 都有自己独立的完整提示词
   - ✅ 每个 Agent 执行时都会独立调用 LLM，使用自己的提示词
   - ✅ Agent 之间通过任务分配和结果返回来协作，而不是共享提示词
这就是华为 DeepDiver-V2 架构的核心设计理念： 通过独立的提示词来控制每个 Agent 的行为，通过任务分配和结果返回来实现多 Agent 协作 ！


# 这个架构确实存在 严重依赖模型推理 的问题：

核心问题 ：

1. ❌ 完全依赖 LLM 的推理能力 ，没有硬编码的规则作为后备
2. ❌ 缺少容错机制 ，模型推理失败时只能返回错误
3. ❌ 缺少验证机制 ，无法检测和纠正模型的错误决策
4. ❌ 提示词敏感度高 ，微小的变化可能导致完全不同的行为
风险表现 ：

- 🔴 "降智"现象 ：模型推理能力下降时，整个系统性能急剧下降
- 🔴 任务失败率高 ：复杂任务可能因为模型推理失败而无法完成
- 🔴 不可预测性 ：相同输入可能产生完全不同的输出
改进方向 ：

- ✅ 引入规则引擎作为后备机制
- ✅ 引入多轮验证机制
- ✅ 引入回退机制
- ✅ 引入多模型集成
这是一个典型的**"Prompt Engineering 架构"的固有弱点**，需要在灵活性和可靠性之间找到平衡！
